락을 관리하는 두 가지 방식
- 읽기 락, 쓰기 락을 둘다 사용 (쓰기 락이 어디서 사용되고 있는 경우 쓰기 잠금이 해제될 때가지 읽을 수 없음) -> 많은 트랜젝션이 필요할 경우 문제가 될 수 있다.
- 쓰기 락만 사용하기 (버전 관리) -> 데이터를 수정하는 데 오래 걸리는 쿼리가 있으면 문제가 될 수 있다.

락 단위로는 테이블, 페이지, 행 잠금 세 범위가 있다. mysql, sql서버는 세 방식을 모두 사용하고 오라클은 행 잠금만 사용한다. 오라클과 달리 mysql은 행 잠금 <=> 페이지로 에스컬레이션 한다.

트랜젝션은 SQL을 묶어서 전부 성공하거나 전부 실패하도록 만드는 장치 (원자성)

<br>

트랜젝션을 관리하는 두 가지 방식
1. 활성 트랜젝션 내에서는 명시적으로 트랜젝션을 시작하고 끝낼 필요가 없다. 하나의 트랜젝션이 끝나면 바로 다른 트랜젝션이 시작된다. -> 오라클?
2. 명시적으로 트랜젝션을 시작하고 끝내는 방식 (각 트랜젝션은 개별적으로 이루어짐)

<br>

commit이나 rollback 이외의 다른 이유로 트랜젝션이 종료되는 경우
- 중간에 서버가 끊김
- alter table 같은 SQL 스키마 구문의 경우 자동으로 현재 트랜젝션을 종료하고 새 트랜젝션을 시작함
- 다른 start transaction 명령이 들어옴
- 교착 상태 (a가 b를 대기, b가 a를 대기) -> 이 경우 한쪽의 트랜젝션을 종료한다.

<br>

csv : 데이터를 쉼표로 구분
InnoDB : 행 수준 잠금 (mysql)

<br>

트랜젝션 세이브포인트

이름을 지정하더라도 commit하지 않으면 반영되지 않는다.
세이브포인트의 이름을 지정하지 않고 rollback하면 모든 세이브포인트가 무시된다.
