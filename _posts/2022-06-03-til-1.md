[1] 인덱스

테이블에 데이터가 순서대로 저장되는 것이 아니라 비어 있는 공간에 들어가기 때문에 원하는 데이터를 찾으려면 테이블을 모두 훑어야 한다.

인덱스는 데이터가 테이블의 어디에 저장되어 있는지 정보를 제공하는 특수한 테이블이다.

열 단위로 지정할 수 있다. 기본 키의 경우 제약조건을 지정하면서 인덱스도 같이 생성?


```sql
show index

create (인덱스 이름)
on (테이블) (열)
```

여러 가지 방식의 인덱스가 있다.

- B tree 인덱스 (부모 노드 - 자식 노드로 이루어져 있음. 이름을 검색한다고 했을 때 알파벳 - 세부 이름 이런 식으로 깊이가 생김. 어떤 하나의 부모에 너무 많은 자식이 생기거나 너무 적은 자식이 남는 일이 없도록 데이터베이스는 계속해서 관리 -> 속도를 보장함)

- 비트맵 인덱스 (0, 1처럼 열에 포함된 값의 종류가 적을 때(가수가 낮은) 사용할 수 있는 인덱스. 기본 키 인덱스처럼 행의 개수에 비해 열에 포함된 값의 종류가 많아지면 많아질수록 사용하기 어려워진다.)

- 텍스트 인덱스 (문서를 읽을 때 사용. 오라클에서는 오라클 텍스트? 사용)

innoDB? (행 잠금) 스토리지 사용해야 함


고유 인덱스를 지정할 수 있다. 
```sql
create unique index
```

다중 열에 대한 인덱스를 만들 수 있다.
```sql
create index date_pymnt
on payment (payment_date, amount)
```

select문으로 일반적인 검색을 할 때 서버는 세 가지 전략 중 하나를 사용할 수 있다.
id, first_name, last_name

- customer 행을 모두 살펴본다.
- last_name 인덱스만 사용해서 행을 확인한 후 first_name과 매치되는지 확인
- last_name 인덱스, first_name 인덱스 둘 다 사용

이것을 확인하려면 select문을 작성하기 전에 explain을 넣어주면 된다.
이것은 쿼리 튜닝의 문제이다.


인덱스도 결국 테이블이기 때문에 어떤 행을 변경/삭제하면 다른 행의 인덱스도 다 업데이트해야 한다는 문제가 생긴다. 따라서 인덱스를 무조건 사용하지 말고 필요한 곳에만 사용해야 한다.

- 기본 키
- 외래 키
- date처럼 검색에 자주 사용되는 열


[2] 제약조건

- 기본 키
- 외래 키
- 고유 (기본 키는 고유 제약조건의 특수한 케이스)
- 체크

테이블을 만들 때 설정할 수 있고 테이블을 만든 후 alter table add constraint로 나중에 추가할 수도 있다.


이중 외래 키 제약조건의 경우
참조한 테이블이 자식 테이블이고 참조당한 테이블이 부모 테이블이라고 할 때, 부모 테이블에서 값을 변경/삭제하면 고아 행이 생길 수 있다.

이 문제를 해결하기 위해

on delete
on update
*
restrict
cascade
set null

이렇게 키워드를 조합해서 사용한다. 이 키워드는 옵션이므로 0, 1, 2개를 선택해서 사용할 수 있다.

restrict : 부모 테이블의 값을 참조하고 있는 자식 테이블이 있는 경우 부모 테이블의 변경을 시도하려고 할 때 Exception을 발생시킨다.

cascade : 부모 테이블의 값을 변경하면 자식 테이블에도 변경사항을 전파시킨다.

set null : 삭제하거나 변경할 때 자식 테이블의 행을 null값으로 수정한다.

=> 고아 레코드가 생기는 것을 막는다.
